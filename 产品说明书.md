## 一、产品概述

### 1.1 产品定位

**智能点菜系统**是一款面向中式餐饮门店的全栈式点餐管理解决方案，覆盖「顾客自助扫码点餐」+「员工端订单管理」+「厨房出品调度」三大核心场景，实现从下单、制作、上菜到结账清台的完整业务闭环。

### 1.2 产品愿景

> 让中小餐饮门店以最低的技术成本，获得媲美连锁品牌的数字化运营能力。

### 1.3 核心价值主张

| 价值维度 | 具体体现 |
|----------|----------|
| **降本增效** | 顾客自助点餐减少服务员工作量；厨房大屏替代纸质单据，降低出错率 |
| **实时协同** | 沽清状态多端同步，避免「点完菜才发现没有了」的尴尬 |
| **数据驱动** | 营业统计、热门菜品、桌位贡献等数据支撑经营决策 |
| **灵活部署** | 单服务器即可运行（PocketBase + Nginx），Docker 一键部署 |

### 1.4 当前版本

| 属性 | 值 |
|------|-----|
| 前端版本 | 1.0.2 |
| 后端 PocketBase | v0.22.27 |
| 单元测试 | 751 用例通过，Branch 覆盖率 **85.41%** |

## 四、核心业务流程

### 4.1 订单全生命周期

```
┌─────────┐    服务员/顾客下单     ┌─────────┐    厨师开始制作     ┌─────────┐
│  空闲    │ ───────────────────→ │ pending │ ─────────────────→ │ cooking │
│ (idle)  │                      │(待确认) │                     │(制作中) │
└─────────┘                      └─────────┘                     └─────────┘
                                      │                              │
                                      │ 取消订单                      │ 制作完成
                                      ▼                              ▼
                               ┌─────────┐                     ┌─────────┐
                               │cancelled│                     │ cooked  │
                               │(已取消) │                     │(待上菜) │
                               └─────────┘                     └─────────┘
                                                                      │
                                                                      │ 服务员上菜
                                                                      ▼
                                                               ┌─────────┐
                                                               │ serving │
                                                               │(上菜中) │
                                                               └─────────┘
                                                                      │
                                                                      │ 全部上菜完毕
                                                                      ▼
                                                               ┌─────────┐
                                                               │ dining  │
                                                               │(用餐中) │
                                                               └─────────┘
                                                                      │
                                                                      │ 顾客结账
                                                                      ▼
                                                               ┌─────────┐
                                                               │completed│
                                                               │(已结账) │
                                                               └─────────┘
                                                                      │
                                                                      │ 顾客离店/清台
                                                                      ▼
                                                               ┌─────────┐
                                                               │ settled │
                                                               │(已清台) │
                                                               └─────────┘
```

### 4.2 状态机规则

| 当前状态 | 可流转至 | 触发条件 |
|----------|----------|----------|
| pending | cooking, cancelled | 厨师点击「开始制作」/ 取消订单 |
| cooking | serving, cancelled | 所有单品变为 cooked / 取消 |
| serving | dining, cancelled | 所有单品变为 served / 取消 |
| dining | completed, cancelled | 顾客结账 / 取消 |
| completed | settled | 服务员确认清台 |
| settled | — | 终态 |
| cancelled | — | 终态 |

**重要约束**：
- 单品状态更新不允许订单整体状态**回退**（如 serving → cooking 被静默忽略）
- 终态订单（completed/settled/cancelled）禁止修改菜品状态或追加菜品
- 后端 Hook 在每次更新时**自动推断**订单整体状态，不信任前端传入的状态值

### 4.3 桌台管理流程

```
订单创建 ──→ 后端 Hook 自动开台 ──→ table_status 变为 dining
                                      │
                                      ├── 记录 currentOrderId
                                      └── 记录 openedAt

订单完成/取消/删除 ──→ 后端 Hook 自动清台 ──→ table_status 变为 idle
                                               └── currentOrderId 清空
```

**占用检查**：
1. 前端：提交前查询 table_status，dining 时 Toast 阻断
2. 后端：onRecordBeforeCreateRequest 中再次检查，原子性阻止入库
3. Node.js 公共 API：createOrder 前同样检查

### 4.4 沽清业务流程

```
员工端标记沽清 ──→  optimistic UI 更新 ──→ 后端更新 dishes.soldOut
      │                                   │
      │ 10s 轮询 / SSE                    │ 顾客端 10s 轮询
      ▼                                   ▼
顾客端显示「已沽清」                 厨房大屏正常显示（不拦截）
不可添加至购物车
```

**自动重置**：每日 04:00 服务器 cron 任务自动清空所有沽清标记。

### 4.5 顾客扫码点餐流程

```
顾客扫码
  │
  ▼
┌─────────────────┐
│  检查桌台状态    │
└─────────────────┘
  │
  ├── 有未完成订单 ──→ 自动加入，显示「追加」模式
  │                      └── 加载已有订单菜品到购物车
  │
  └── 无订单 ──→ 显示人数选择模态框
       │
       ▼
  ┌─────────────────┐
  │  浏览菜品 → 添加购物车 → 提交订单  │
  └─────────────────┘
       │
       ▼
  保存 accessToken 到 sessionStorage
       │
       ▼
  显示成功提示，可继续追加
```

|------|------|------|------|
| 前端框架 | Vue | 3.5 | 组件化 + Composition API |
| 构建工具 | Vite | 8 | 开发服务器 + 生产构建 |

| E2E 测试 | Playwright | ~1.49 | 端到端自动化测试 |

| 公共 API | Fastify | 5 | 顾客端无认证 API |
| 后端钩子 | Goja (JS VM) | — | 业务规则钩子 |

### 5.3 关键架构决策

| 决策 | 说明 | 权衡 |
|------|------|------|
| **PocketBase 作为主后端** | 单二进制部署，内置 SQLite/认证/文件存储/实时订阅 | 简化了运维，但 SQLite 并发性能有上限，不适合超大规模 |
| **双 API 层** | 员工端直连 PocketBase（JWT 认证），顾客端走 Fastify（Token 认证） | 安全隔离，但增加了系统复杂度 |
| **后端 Hook 执行业务规则** | 金额计算、状态推断、桌台同步全部在 PocketBase Hook 中执行 | 前端不可信，数据一致性由后端保证 |
| **Denormalized table_status** | 独立集合维护桌台占用状态 | O(1) 查询，但需 Hook 保证同步 |
| **Integer Cents 金额计算** | 所有金额以「分」为单位计算 | 避免浮点精度问题 |
| **accessToken 顾客会话** | 无需注册，随机 Token 即可访问订单 | 用户体验好，但 Token 泄露风险需关注 |

### 6.2 状态枚举

```typescript
// 订单整体状态
pending   → 待确认
cooking   → 制作中
serving   → 上菜中
dining    → 用餐中
completed → 已结账
settled   → 已清台
cancelled → 已取消

pending → 待制作
cooking → 制作中
cooked  → 已制作（待上菜）
served  → 已上菜
```

## 七、安全与合规

### 7.1 安全机制

| 层级 | 措施 |
|------|------|
| **XSS 防护** | `escapeHtml()` 转义所有用户输入；`setSafeHtml()` 白名单过滤；禁止 `innerHTML` 直接拼接 |
| **Filter 注入防护** | `escapePbString()` 转义单引号、操作符等；`sanitizePbLike()` 过滤非法字符 |
| **金额防篡改** | 后端 Hook 强制重算金额，不信任前端传入的 totalAmount/finalAmount |
| **桌台占用原子性** | 前端预检 + 后端 Hook 二次检查 + Node.js API 三重保障 |
| **状态机防回退** | 后端 Hook 维护状态优先级，低优先级状态不覆盖高优先级 |
| **已做菜不可删** | 后端 Hook 拦截删除 cooking/cooked/served 的菜品 |
| **终态不可追加** | 后端 Hook 拦截对 completed/settled/cancelled 订单的菜品追加 |
| **Token 验证** | 顾客端 accessToken 校验，防止越权访问他人订单 |
| **Rate Limit** | Fastify 公共 API 限流（1 分钟 100 次） |
| **CSP/Helmet** | Fastify 注册 helmet 中间件 |

### 7.2 数据权限

- 员工端：JWT Bearer Token，PocketBase 集合规则 `@request.auth.id != ''`
- 顾客端：无需登录，通过 accessToken 访问特定订单
- 厨房大屏：公开路由，无需认证

## 九、待优化事项（产品经理洞察）

### 9.1 高优先级（直接影响营收或体验）

#### 🔴 P0-1: 会员与复购体系（缺失）

**现状**：系统无用户体系，顾客每次扫码都是匿名状态，无法识别回头客、无法积累消费数据。

**影响**：
- 无法做会员营销（积分、优惠券、生日特权）
- 无法分析顾客生命周期价值（LTV）
- 无法推送「您常点的菜」提升转化率

**建议方案**：
- 短期：顾客端增加「手机号留座」功能，记录 customerPhone
- 中期：接入微信/支付宝小程序授权，获取 OpenID
- 长期：独立会员系统，支持积分、储值、优惠券

#### 🔴 P0-2: 支付闭环（缺失）

**现状**：系统只展示收款二维码图片，无法直接完成支付；订单状态需服务员手动点击「确认结账」推进。

**影响**：
- 高峰期服务员需频繁手动操作，增加工作量
- 无法自动对账（系统金额 vs 实际收款金额）
- 顾客无法在线支付（只能扫码转账）

**建议方案**：
- 接入微信支付/支付宝当面付 API
- 支付成功后自动推进订单状态 completed → settled
- 增加「支付中」临时状态，防止重复支付

#### 🔴 P0-3: 多规格菜品支持（缺失）

**现状**：菜品只有单一价格，不支持「大份/小份」「辣度选择」等规格。

**影响**：
- 火锅店、烧烤店等场景无法使用（锅底分麻辣/清汤/鸳鸯）
- 只能拆分为多个菜品（如「锅底-麻辣」「锅底-清汤」），菜单冗长

**建议方案**：
- dishes 集合增加 `specs` JSON 字段：`[{name: "大份", price: 128}, {name: "小份", price: 88}]`
- 顾客端/员工端选择菜品时弹出规格选择
- 订单 items 中记录 `specName` + `specPrice`

#### 🔴 P0-4: 库存管理（缺失）

**现状**：沽清是手动标记的，系统不知道菜品实际库存。沽清后一键清空，第二天需要重新检查库存再手动标记。

**影响**：
- 每天开业需人工盘点哪些菜品已售罄
- 沽清标记可能遗漏，导致顾客下单后无法出餐
- 无法预警「某菜品剩余 5 份」

**建议方案**：
- dishes 集合增加 `stock` 字段（日库存量）
- 后端 Hook 在创建/更新订单时自动扣减库存
- 库存 ≤ 0 时自动标记 soldOut
- 每日 04:00 cron 重置库存（从预设值）

### 9.2 中优先级（提升效率或体验）

#### 🟡 P1-1: 服务员 App / 小程序

**现状**：员工端是 Web 页面，厨房大屏也是 Web 页面。服务员需要带手机 + 前台 PC，或厨房需要专门放一台平板/电脑。

**建议方案**：
- 将员工端打包为 PWA，支持离线缓存、桌面图标
- 或开发微信小程序员工端（扫码登录）
- 厨房大屏可适配为竖屏模式，更适合挂墙展示

#### 🟡 P1-2: 预订与排号系统

**现状**：系统只支持「到店后」下单，不支持提前预订。

**建议方案**：
- 新增「预订」功能：顾客选择日期/时间/人数/桌号，预交定金
- 预订到店后一键转正式订单
- 高峰期排号叫号（微信小程序推送叫号通知）

#### 🟡 P1-3: 后厨分档口

**现状**：厨房大屏展示所有菜品，但大型餐厅通常有「凉菜档」「热菜档」「面点档」多个厨房。

**建议方案**：
- dishes 增加 `kitchenStation` 字段（档口）
- KDS 支持按档口筛选展示
- 每个档口独立的大屏/打印机

#### 🟡 P1-4: 外卖/自提订单

**现状**：系统只支持堂食（必须选择桌号）。

**建议方案**：
- 订单增加 `orderType`: dine-in / takeout / delivery
- 外卖订单无需桌号，需填写地址/电话
- 对接美团/饿了么开放平台（长期）

#### 🟡 P1-5: 数据看板增强

**现状**：统计页面只有基础图表，缺乏深度分析。

**建议方案**：
- 增加「翻台率」指标：平均用餐时长、各时段翻台次数
- 增加「菜品毛利分析」：需录入菜品成本价
- 增加「服务员绩效」：各服务员下单数量/金额
- 增加「异常订单」：取消率、退菜率分析
- 支持数据导出为 PDF 日报/月报

### 9.3 低优先级（体验优化）

#### 🟢 P2-1: 顾客端 UI 升级

- 增加菜品图片展示（当前只有文字）
- 增加「推荐菜品」「本店招牌」标签
- 支持套餐组合（如「双人套餐」包含多道菜品）
- 下单后实时查看菜品制作进度（类似外卖 App 的「商家已接单→制作中→已上菜」）

#### 🟢 P2-2: 语音叫号

- 菜品制作完成后，厨房可点击「叫号」，前台语音播报「A1 桌菜品已做好，请上菜」
- 适用于大型餐厅，避免服务员频繁查看 KDS

#### 🟢 P2-3: 智能推荐

- 基于历史订单数据，推荐「点了这道菜的人也点了...」
- 基于时段推荐（午餐推荐商务套餐，晚餐推荐聚会大菜）

#### 🟢 P2-4: 多语言支持

- 当前系统仅支持中文，可扩展支持英文/日文等（面向外国游客场景）
