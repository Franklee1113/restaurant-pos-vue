# 全景餐桌管理页面 — 业务逻辑深度评审报告

> 评审维度：业务场景 × 状态机 × 系统逻辑 | 日期：2026-04-22

---

## 一、评审方法

以「订单状态机」和「桌台生命周期」为两条主线，逐一检验每个业务场景下方案的表现。覆盖 7 个订单状态 × 4 个菜品状态 × 3 个主要业务角色（店长/服务员/厨师视角）。

---

## 二、系统状态机回顾

### 2.1 订单状态机（7 个状态）

```
pending ──► cooking ──► serving ──► dining ──► completed ──► settled
    │           │           │          │            │
    └───────────┴───────────┴──────────┴────────────┘
                          │
                          ▼
                      cancelled
```

| 状态 | 业务含义 | 触发方式 |
|---|---|---|
| `pending` | 待确认 | 订单创建后初始状态 |
| `cooking` | 制作中 | Hook 推断：有任何菜品 cooking |
| `serving` | 上菜中 | Hook 推断：全部菜品 cooked 但未全部 served |
| `dining` | 用餐中 | Hook 推断：全部菜品 served |
| `completed` | 已结账 | 手动点击"结账" |
| `settled` | 已清台 | 清台操作后自动流转 |
| `cancelled` | 已取消 | 手动取消 |

### 2.2 菜品状态机（4 个状态）

```
pending ──► cooking ──► cooked ──► served
```

由 KDS `updateOrderItemStatus` 驱动，逐菜品推进。

### 2.3 桌台状态机（2 个真实状态）

```
idle ◄──────► dining
```

- `idle`：空闲，无当前订单
- `dining`：占用，有 currentOrderId 指向当前订单
- `pending_clear`：类型定义中有，但 **Hook 从未设置**，属于"幽灵状态"

**关键事实**：订单 `completed` 时 table_status **仍为 dining**，直到清台才变 `idle`。

## 三、全场景业务推演

### 场景 A：新订单刚创建

**系统状态**：
- 订单：`status=pending`, `items=[{status:pending}×N]`
- 桌台：`status=dining`, `currentOrderId=order.id`

**桌台全景表现**：
- 卡片显示：🔵 待确认
- 等待时长：0~5 分钟（灰色）
- 菜品：全部 ⚪ 待制作

**业务合理性**：✅ 合理。服务员能看到"新单来了，还没开始做"。

---

### 场景 B：厨师开始制作（部分菜品）

**系统状态**：
- 订单：`status=cooking`（Hook 推断：anyCooking=true）
- 桌台：`status=dining`
- 菜品 A：`status=cooking`，其他：`status=pending`

**桌台全景表现**：
- 卡片显示：🔵 制作中
- 菜品 A：🔵 制作中（脉冲动画），其他：⚪ 待制作

**业务合理性**：✅ 合理。服务员可以据此向顾客解释"铁锅鱼正在做，锅底还没开始"。

---

### 场景 C：全部菜品做好，等待上菜

**系统状态**：
- 订单：`status=serving`（Hook 推断：allDone=true, allServed=false）
- 桌台：`status=dining`
- 全部菜品：`status=cooked`

**桌台全景表现**：
- 卡片显示：🟣 上菜中
- 全部菜品：🟢 已做好

**业务合理性**：✅ 合理。这是服务员最该关注的场景——"菜都好了，快上！"

**⚠️ 发现**：当前系统没有"上菜"操作来把 `cooked` → `served`。菜品状态只能由 KDS 的"开始制作"和"已完成"两个按钮推进。

实际上 `cooked` → `served` 的流转在哪发生？

检查代码：`updateOrderItemStatus` 支持 `served` 状态，但 KDS 只有"开始制作"（→ cooking）和"已完成"（→ cooked）两个按钮。没有"已上菜"按钮。

**结论**：`cooked` → `served` 的流转在当前系统中**没有 UI 入口**。这意味着：
- 菜品做好后（cooked），服务员无法标记"已上菜"
- Hook 永远无法推断出 `allServed=true`
- 订单状态永远无法从 `serving` 自动推进到 `dining`
- 订单状态需要手动从 `serving` → `dining` 或等待其他机制

这是一个**现有系统的业务缺陷**，不是桌台全景页面的问题。但桌台全景页面会真实地反映出"菜品已做好但没上菜"的状态，这是有价值的。

---

### 场景 D：上菜完成，顾客用餐

**系统状态**：
- 订单：`status=dining`（Hook 推断：allServed=true）
- 桌台：`status=dining`
- 全部菜品：`status=served`

**⚠️ 但现实是**：由于上述 `cooked→served` 无 UI 入口的问题，实际订单可能卡在 `serving` 状态，由服务员手动点击流转到 `dining`。

**桌台全景表现**：
- 卡片显示：🟠 用餐中
- 全部菜品：⚪ 已上菜（淡化 opacity-50）

**业务合理性**：✅ 合理。此时服务员/店长知道"这桌在吃了，不用管"。

---

### 场景 E：结账完成，待清台

**系统状态**：
- 订单：`status=completed`（手动点击"结账"）
- 桌台：`status=dining`（注意：仍为 dining！）
- 全部菜品：`status=served`（或 `cooked`，取决于实际上菜流程）

**桌台全景表现**：
- 卡片显示：🟡 待清台（复合判断：ts.dining + order.completed）
- 等待时长：持续增加（反映"结账后占用多久"）
- 全部菜品：淡化显示

**业务合理性**：✅ 合理。店长能看到"这桌已经结账了，该收拾了"。

**但存在一个现实问题**：顾客结账后不一定立刻离开。等待时长变红（≥20分钟）时，服务员可以去礼貌询问是否需要清台。这是有用的业务指标。

---

### 场景 F：顾客追加菜品

**系统状态变化**：
1. 原订单：`status=dining`，全部 served
2. 追加 2 道菜品：`items` 追加两个 `{status:pending}`
3. Hook 推断：`inferred=pending`，但 `statusPriority[pending]=0 < statusPriority[dining]=3`
4. **Hook 静默忽略回退**，订单状态 **保持 dining**

**桌台全景表现**：
- 卡片显示：🟠 用餐中（订单状态仍为 dining）
- 追加菜品：⚪ 待制作（显示在列表末尾）

**⚠️ 业务问题**：
- 订单状态没有回退到 cooking，意味着 KDS 的声音提示逻辑 `hadNewPending` 可能检测不到（因为订单状态没变，只是 items 增加了）
- 厨房可能在忙其他订单，不知道这桌有新追加
- 但桌台全景页面会**正确显示追加的 pending 菜品**，服务员可以看到后去催菜

**结论**：桌台全景在追加场景中弥补了 KDS 通知不到的问题，**业务价值显著**。

---

### 场景 G：清台完成

**系统状态**：
- 订单：`status=settled`
- 桌台：`status=idle`, `currentOrderId=''`

**桌台全景表现**：
- 该卡片**消失**（因为只展示 dining 桌台）

**业务合理性**：✅ 合理。清台后不需要再关注。

---

### 场景 H：全店空闲

**系统状态**：
- 所有桌台：`status=idle`
- 无活跃订单

**桌台全景表现**：
- 空状态："🎉 所有桌台空闲"

**业务合理性**：✅ 合理。店长一眼就知道"现在没客人"。

## 四、跨角色业务价值评估

| 角色 | 核心需求 | 方案满足度 | 价值 |
|---|---|---|---|
| **店长** | 一扫全店状态，找瓶颈桌 | ⭐⭐⭐⭐⭐ | 等待时长颜色提示直接暴露服务延误 |
| **服务员** | 知道哪桌该上菜、哪桌该催菜 | ⭐⭐⭐⭐⭐ | cooked 菜品绿色提示"快上"，pending 灰色提示"还没做" |
| **收银员** | 知道哪桌已结账待清台 | ⭐⭐⭐⭐⭐ | 待清台黄色卡片一眼识别 |
| **迎宾/排号** | 知道有多少空桌 | ⭐⭐⭐ | 统计栏显示空闲数量，但看不到具体哪张空 |

**结论**：方案对不同角色的业务价值都很高。

## 五、现有系统缺陷对桌台全景的影响

### 缺陷 1：`cooked → served` 无 UI 入口

**影响**：
- 菜品做好后无法标记"已上菜"
- 订单状态卡在 `serving`，不会自动到 `dining`
- 桌台全景中会出现"🟣 上菜中"状态长期不变的情况

**应对方案（在桌台全景中）**：
- 显示 `cooked` 菜品为 🟢 已做好，提醒服务员"该上菜了"
- 这是现有系统下能做到的最佳体验
- **建议后续迭代**：增加"标记上菜"按钮或批量上菜功能

### 缺陷 2：追加菜品后订单状态不回退

**影响**：
- 订单保持 `dining`，KDS 可能不播放新订单提示音
- 厨房可能忽略追加菜品

**桌台全景的价值**：
- 追加的 pending 菜品在卡片上清晰显示
- 服务员可以看到后主动催菜
- **桌台全景成为追加菜品的"兜底提醒"渠道**

### 缺陷 3：`table_status` 与 `orders` 数据不一致

**场景**：订单被手动删除、Hook 执行失败、数据修复等。

**桌台全景的防御**：
- `Map<tableNo, Order[]>` 结构天然支持一桌多单检测
- 无匹配订单时显示异常提示（而非崩溃）

## 六、统计栏"金额"口径的修正建议

原设计建议统计栏显示"今日营收"。从业务角度分析：

| 口径 | 问题 |
|---|---|
| 今日营收（00:00~now） | 需要查询 settled 订单，但 settled 订单不在活跃订单中，需要额外查询 |
| 当前营业中金额 | 只累加显示中的卡片（占用中 + 待清台），口径清晰，实现简单 |

**建议**：统计栏显示 **"当前营业中 ¥X,XXX"** 而非"今日营收"。

理由：
1. 实现简单，不需要额外 API 调用
2. 店长关心的是"现在正在产生的营业额"
3. 避免 completed→settled 快速流转导致数字跳变

## 七、快捷操作按钮的业务合理性

| 按钮 | 在哪些订单状态显示 | 业务合理性 |
|---|---|---|
| **查看** | 全部 | ✅ 跳转详情页，查看完整信息 |
| **编辑** | pending / cooking / serving / dining | ✅ 修改菜品、人数、折扣等 |
| **编辑** | completed | ❌ 已结账不可编辑，应隐藏/禁用 |
| **打印账单** | 全部 | ✅ 任何阶段都可以打印 |
| **清台** | completed（待清台） | ✅ 清台是待清台桌的核心操作 |

**修正**：`completed` 状态的桌台隐藏「编辑」按钮。

---

## 八、卡片点击行为的业务合理性

**建议**：卡片整体可点击 → 跳转 `OrderDetailView`
- 服务员/店长最常用的操作是"查看详情"
- 快捷操作按钮（查看/编辑/打印/清台）阻止事件冒泡
- 和 OrderListView 的交互习惯保持一致（点击行查看详情）

---

## 九、追加菜品单独显示的业务验证

由于已实施追加不合并方案（Solution A），同一 dishId 在 items 中可能出现多次：

```
● 铁锅鱼    ×1  ¥68
  🔵 制作中

● 铁锅鱼    ×1  ¥68    ← 追加的
  ⚪ 待制作
```

**业务合理性**：✅ 极高。

服务员可以清晰告知顾客：
- "您第一份铁锅鱼正在做"
- "第二份还没开始做，大概还要等 15 分钟"

如果合并显示（老方案）：
- "铁锅鱼 ×2，制作中" → 顾客以为两份都在做，实际第二份还没开始

**结论**：追加不合并方案 + 桌台全景独立显示，组合效果最佳。

## 十、最终评审结论

### ✅ 方案在业务逻辑上完全合理

1. **状态机覆盖完整**：7 个订单状态 × 4 个菜品状态，每个组合都有合理的显示方式
2. **弥补了现有系统缺陷**：追加菜品不回退的问题、cooked→served 无入口的问题，在桌台全景中都能被服务员发现
3. **多角色价值高**：店长/服务员/收银员都能从这个页面获得关键信息
4. **业务操作流程顺畅**：查看→编辑→打印→清台，和现有系统操作流程一致

### ⚠️ 需微调的细节（不影响整体方案）

| 2 | completed 桌台显示「编辑」按钮 | 隐藏/禁用 |
| 3 | `cooked→served` 无系统入口 | 方案中已用 🟢"已做好"提醒服务员，后续可迭代增加"标记上菜"功能 |
