# 功能开发完成自检清单

> **用途**: 每个功能开发完成后，开发者必须逐项自检，避免"代码写了但功能不工作"的低级遗漏
> **更新日期**: 2026-04-20

---

## 一、数据加载验证（最高优先级）

- [ ] **页面挂载时，所有需要的数据源都已加载**
  - 检查 `onMounted` / `setup` 中是否调用了所有必要的 API（不要只调了订单接口，忘了菜品接口）
  - 检查加载失败时是否有降级处理（空状态、错误提示、静默忽略）
  - 反例：SoldOutDrawer 用了 `allDishes` 但 `onMounted` 没调 `DishAPI.getDishes()`

- [ ] **组件接收的 props / 注入的数据非空**
  - 子组件依赖父组件传入的数据时，确认父组件确实传了
  - 列表/表格组件：空数组时是否显示 EmptyState，而非空白或报错

---

## 二、UI / 交互验证

- [ ] **所有新增按钮/入口点击后，行为符合预期**
  - 按钮是否存在、文案是否正确
  - 点击后是否打开正确的页面/弹窗/抽屉
  - 点击后是否发起了预期的 API 请求

- [ ] **禁用/只读状态正确**
  - `completed` / `settled` 等终态是否禁用了编辑按钮
  - 加载中状态是否禁用了提交按钮（防止重复提交）
  - 无权限时是否隐藏了操作入口

- [ ] **表单提交有前置校验**
  - 必填项是否拦截
  - 数值范围是否校验（如折扣 0-10）
  - 提交前是否有确认弹窗（删除、清台等危险操作）

---

## 三、边界状态验证

- [ ] **空状态**
  - 列表为空时是否显示 EmptyState
  - 搜索无结果时是否友好提示
  - 图表无数据时是否显示占位符

- [ ] **异常状态**
  - 网络断开时：是否有重试机制或离线提示
  - API 返回 401/403/500 时：是否跳转登录页或 Toast 提示
  - 并发操作（两个人同时编辑同一订单）是否有冲突处理

- [ ] **终态订单处理**
  - `completed`（已结账）：不可编辑、不可追加菜品、可清台
  - `settled`（已清台）：不可编辑、不可追加菜品、不可清台
  - `cancelled`（已取消）：不可追加菜品

---

## 四、多端一致性验证

- [ ] **员工端（管理后台）**
  - 数据是否正确显示
  - 操作是否生效
  - 状态变更后是否实时同步（SSE / 轮询 / 刷新）

- [ ] **顾客端（扫码点餐）**
  - 员工端的变更（如沽清）是否同步到顾客端
  - 顾客下单后，员工端是否收到通知

- [ ] **KDS（厨房大屏）**
  - 新订单是否出现
  - 状态变更（pending → cooking → served）是否同步

---

## 五、测试验证

- [ ] **单元测试覆盖新增逻辑**
  - 是否为新函数/composable 编写了测试
  - 是否为新增视图编写了测试（特别是数据加载和交互）
  - 边界条件是否有测试（空数组、错误响应、非法状态流转）

- [ ] **测试能通过**
  - `npm run test:unit` 全部通过
  - `npm run type-check` 无错误（不要跳过类型检查）

- [ ] **E2E 关键路径**
  - 如果修改了核心流程（登录→建单→状态变更→结账→清台），是否更新了 Playwright 测试

---

## 六、部署前验证

- [ ] **本地构建通过**
  - `npm run build` 成功（含类型检查）
  - 不要只用 `npm run build-only` 绕过类型检查

- [ ] **部署后验证**
  - 部署脚本执行后，curl 生产环境确认 JS 文件名与本地构建产物一致
  - 打开生产环境页面，手动走一遍功能主流程
  - 检查浏览器控制台是否有报错

---

## 七、常见遗漏案例（反面教材）

| 功能 | 遗漏项 | 后果 |
|------|--------|------|
| SoldOutDrawer | `onMounted` 没加载 `allDishes` | 抽屉打开为空，已沽清菜品看不到 |
| 编辑订单阻断 | 只改了 `OrderDetailView`，忘了 `OrderListView` | 列表页仍可编辑已结账订单 |
| 部署脚本 | `build-only` 跳过类型检查 | 类型错误被带到生产环境 |
| Nginx 配置 | root 目录与 deploy.sh 目标不一致 | "假部署"——脚本成功但用户看到旧版本 |
| 状态机升级 | 只改了后端 Hook，前端状态标签未同步 | 前后端状态语义不一致 |

---
