## 二、P0 - 架构级缺陷（必须修正）

### P0-1: 乐观更新的并发竞争条件

**问题描述**:

```typescript
// OrderFormView.vue
async function markDishSoldOut(dish: Dish) {
  const idx = dishes.value.findIndex((d) => d.id === dish.id)
  dishes.value[idx] = { ...dish, soldOut: true, soldOutAt: new Date().toISOString() }  // ← 乐观更新
  await DishAPI.toggleSoldOut(dish.id, true)  // ← API 请求
  // 如果在这之间另一个用户/标签页也修改了这道菜...
}
```

**风险场景**:
- 服务员 A 在电脑 1 标记「铁锅鱼」沽清
- 同时服务员 B 在电脑 2 修改「铁锅鱼」的价格
- A 的乐观更新覆盖了 B 的价格修改（或者 B 的更新覆盖了 A 的 soldOut）
- 更致命的是：A 失败后回滚，但回滚的是 A 本地的 snapshot，可能覆盖掉 B 的合法更新

**根因分析**:
PocketBase 的 PATCH 是**整记录覆盖**（非字段级增量更新）。当前 `DishAPI.toggleSoldOut` 只 PATCH `{ soldOut, soldOutNote, soldOutAt }`，但 PocketBase 的 REST API 默认行为是**合并更新**，不是覆盖。然而，如果前端先读取整记录、修改一个字段、再 PATCH 回去，这就是经典的 "read-modify-write" 竞争。

**当前代码实际行为**：
```typescript
// DishAPI.toggleSoldOut 只 PATCH 三个字段
body: JSON.stringify({ soldOut, soldOutNote, soldOutAt })
```
PocketBase 的 PATCH 是字段级合并，所以这个特定调用不会覆盖其他字段。**但是**，乐观更新回滚时：
```typescript
dishes.value[idx] = originalState  // originalState 是本地读取时的快照
```
如果 SSE 在这之间推送了更新的数据，回滚会把 SSE 推送的新数据也覆盖掉。

**修正方案**:

```typescript
// 1. 乐观更新时只修改目标字段，不要替换整个对象
const idx = dishes.value.findIndex((d) => d.id === dish.id)
if (idx !== -1) {
  dishes.value[idx] = { ...dishes.value[idx], soldOut: true, soldOutAt: new Date().toISOString() }
}

// 2. 回滚时只恢复 soldOut 相关字段，不要还原整个 dish
// ❌ 不要这样：dishes.value[idx] = originalState
dishes.value[idx] = { ...dishes.value[idx], soldOut: originalState.soldOut, soldOutAt: originalState.soldOutAt }

// 3. 更好的做法：使用服务端返回的数据作为最终状态
const updated = await DishAPI.toggleSoldOut(dish.id, true)
dishes.value[idx] = { ...dishes.value[idx], ...updated }
```

**优先级**: 🔴 **阻塞开发**

### P0-2: `apiCache` 单实例内存缓存的分布式陷阱

**问题描述**:

```typescript
// src/api/pocketbase.ts
export const apiCache = new MemoryCache()  // ← 单例，进程内内存

async toggleSoldOut(id, soldOut, note) {
  // ...
  apiCache.clear('dishes:all')   // ← 只清除了当前进程的缓存
}
```

**风险场景**:
当前系统是单实例部署（一台服务器），所以这个问题暂时不会触发。但如果未来：
- 使用 PM2 多进程部署 Node.js 公共服务
- 使用多台服务器做负载均衡
- 前端启用了 Service Worker 缓存

`apiCache.clear('dishes:all')` 只影响**当前浏览器标签页**的内存。其他标签页、其他设备、其他进程都不会收到缓存失效通知。

**根因分析**:
`apiCache` 是一个简单的 `Map` 内存缓存，没有任何分布式失效机制。在小型餐厅单设备场景下确实够用，但方案文档中明确提到 "3 秒内所有在线设备感知"，这与缓存设计存在隐性矛盾。

**修正方案**:

**方案 A（推荐，保持简单）**: 完全移除 `DishAPI.getDishes()` 的缓存
```typescript
async getDishes(): Promise<ListResult<Dish>> {
  // 移除缓存，直接请求
  const url = `${PB_URL}/collections/${COLLECTION_DISHES}/records?perPage=100&sort=name`
  const res = await privateRequest<ListResult<Dish>>(url)
  if (!res) throw new APIError('获取菜品列表失败', 500)
  return res  // 不缓存
}
```
> 理由：66 道菜，每道约 200 字节，响应体约 13KB，Gzip 后约 3KB。即使每秒请求 10 次，带宽消耗也极小。缓存带来的收益远低于数据不一致的风险。

**方案 B（保留缓存，增加版本号机制）**:
```typescript
// 全局菜品版本号，toggleSoldOut 时递增
let dishesVersion = 0

async toggleSoldOut(id, soldOut, note) {
  const res = await privateRequest<...>(...)
  apiCache.clear('dishes:all')
  dishesVersion++
  localStorage.setItem('dishes_version', String(dishesVersion))
  return res
}

// 加载时检查版本号
async getDishes() {
  const cached = apiCache.get('dishes:all')
  const localVer = Number(localStorage.getItem('dishes_version') || '0')
  const cacheVer = cached?._version || 0
  if (cached && cacheVer >= localVer) return cached
  // ...fetch
}
```

**优先级**: 🟠 **建议修正**（单实例下暂不触发，但架构上不健康）

### P0-3: PocketBase Hook 中的 N+1 查询性能灾难

**问题描述**:

```javascript
// pb_hooks/orders.pb.js
function validateItemsSoldOut(items) {
  if (!items || items.length === 0) return
  for (let i = 0; i < items.length; i++) {
    const dishId = items[i].dishId
    try {
      const dish = $app.dao().findRecordById('dishes', dishId)  // ← N 次查询！
      if (dish && dish.get('soldOut') === true) {
        throw new Error('...')
      }
    } catch (e) { ... }
  }
}
```

**风险场景**:
一个 20 人的聚餐订单，点了 25 道菜。Hook 执行时需要进行 **25 次** `findRecordById` 查询。PocketBase 基于 SQLite，虽然单表查询很快，但在午市高峰（每秒 5-10 笔订单创建），这些查询会累积成明显的性能瓶颈。

**量化分析**:
- SQLite 单条 `SELECT` 约 0.1-0.5ms
- 25 道菜 × 0.3ms = 7.5ms
- 加上 Go-JS VM 的上下文切换开销，实际可能 15-30ms
- 如果同时有 10 个订单在创建，排队延迟可能达到 150-300ms
- 顾客端扫码下单等待时间从 200ms 增加到 500ms+，体验明显下降

**修正方案**:

```javascript
// 批量查询，一次数据库往返
function validateItemsSoldOut(items) {
  if (!items || items.length === 0) return
  
  // 去重 dishId
  const dishIdSet = new Set()
  for (let i = 0; i < items.length; i++) {
    dishIdSet.add(items[i].dishId)
  }
  const dishIds = Array.from(dishIdSet)
  if (dishIds.length === 0) return

  // 批量查询：使用 IN 语句
  const placeholders = dishIds.map((_, idx) => '{:id' + idx + '}').join(',')
  const filter = 'id in (' + placeholders + ') && soldOut = true'
  
  const params = {}
  for (let i = 0; i < dishIds.length; i++) {
    params['id' + i] = dishIds[i]
  }

  try {
    const soldOutRecords = $app.dao().findRecordsByFilter('dishes', filter, '', 100, 0, params)
    if (soldOutRecords && soldOutRecords.length > 0) {
      const names = []
      for (let i = 0; i < items.length; i++) {
        for (let j = 0; j < soldOutRecords.length; j++) {
          if (items[i].dishId === soldOutRecords[j].id) {
            names.push(items[i].name)
            break
          }
        }
      }
      throw new Error('菜品 ' + names.join('、') + ' 已售罄，无法下单')
    }
  } catch (e) {
    if (e.message && e.message.indexOf('已售罄') !== -1) throw e
    console.error('validateItemsSoldOut error:', e)
  }
}
```

**优先级**: 🔴 **阻塞开发**（高峰期性能影响显著）

### P0-4: SSE 订阅 `dishes` 集合的连接数膨胀

**问题描述**:

方案中要求 `OrderFormView`、`CustomerOrderView` 等多个视图都订阅 `subscribeToDishes`。每个视图打开一个 SSE 连接，如果用户打开多个标签页，连接数线性增长。

**PocketBase 限制**:
- PocketBase 免费版/开源版没有严格的连接数限制，但每个 SSE 连接都是一个活跃的 HTTP 长连接
- 每个连接占用一个 Go routine + 网络句柄
- 如果一个服务员在前台开了 3 个标签页（订单列表、点菜页、订单详情），就是 3 个 SSE 连接
- 加上顾客端的连接，高峰期连接数可能达到 20-50 个

**根因分析**:
当前 `subscribeToOrders` 只有一个视图（KitchenDisplayView）使用，连接数可控。但 `subscribeToDishes` 要在多个视图中使用，连接数会成倍增长。

**更深层问题**：PocketBase 的 SSE 是**按 collection + filter** 订阅的。如果多个标签页订阅同一个 collection（ dishes ），PocketBase 会为每个连接独立推送消息，造成**服务端消息重复推送**。

**修正方案**:

**方案 A（推荐）：共享单例 SSE 连接**

```typescript
// src/api/pocketbase.ts
let sharedDishesUnsubscribe: (() => void) | null = null
let sharedDishesCallbacks = new Set<(record: Dish) => void>()

export async function subscribeToDishes(
  onUpdate: (record: Dish) => void,
): Promise<() => void> {
  // 注册回调
  sharedDishesCallbacks.add(onUpdate)

  // 首次订阅时建立连接
  if (!sharedDishesUnsubscribe) {
    const unsub = await createDishesSSEConnection((record) => {
      sharedDishesCallbacks.forEach((cb) => cb(record))
    })
    sharedDishesUnsubscribe = unsub
  }

  // 返回注销函数
  return () => {
    sharedDishesCallbacks.delete(onUpdate)
    if (sharedDishesCallbacks.size === 0 && sharedDishesUnsubscribe) {
      sharedDishesUnsubscribe()
      sharedDishesUnsubscribe = null
    }
  }
}
```

**方案 B（降级为轮询为主）**:
小型餐厅场景下，SSE 的 ROI 极低。建议：
- 默认使用 10 秒轮询
- SSE 作为可选增强，只在 KitchenDisplayView（必须实时）启用
- 其他视图（点菜页、订单列表）用 `visibilitychange` + 轮询即可

**优先级**: 🔴 **阻塞开发**（连接数膨胀可能导致 PocketBase 响应变慢甚至崩溃）

### P1-2: 长按交互缺少浏览器兼容性处理

**问题描述**:

方案中设计了长按/右键交互：
```vue
<div
  @contextmenu.prevent="!dish.soldOut && openDishMenu(dish)"
  @touchstart.passive="handleTouchStart(dish)"
  @touchend.passive="handleTouchEnd"
>
```

但**完全没有处理以下移动端浏览器默认行为**：
- iOS Safari 长按会弹出「复制/分享」系统菜单
- Android Chrome 长按会触发文字选择
- 部分浏览器长按会触发 `contextmenu` 事件
- 长按 600ms 期间没有视觉反馈，用户不知道是否触发成功

**修正方案**:

```vue
<template>
  <div
    :class="[
      'relative rounded-xl p-3 text-center border transition-all select-none',
      longPressActive ? 'scale-95 bg-gray-200' : '',
      dish.soldOut ? 'bg-gray-100 opacity-60' : 'bg-gray-50'
    ]"
    style="-webkit-touch-callout: none; -webkit-user-select: none; user-select: none;"
    @contextmenu.prevent="handleContextMenu(dish)"
    @touchstart.passive="handleTouchStart(dish)"
    @touchend.passive="handleTouchEnd"
    @touchmove.passive="handleTouchMove"
  >
</template>

<script setup>
const longPressActive = ref(false)
let longPressTimer = null
const LONG_PRESS_DURATION = 600

function handleTouchStart(dish) {
  if (dish.soldOut) return
  longPressActive.value = true
  longPressTimer = setTimeout(() => {
    longPressActive.value = false
    openDishMenu(dish)
    // 触发震动反馈（如果支持）
    if (navigator.vibrate) navigator.vibrate(50)
    longPressTimer = null
  }, LONG_PRESS_DURATION)
}

function handleTouchEnd() {
  longPressActive.value = false
  if (longPressTimer) {
    clearTimeout(longPressTimer)
    longPressTimer = null
  }
}

function handleTouchMove() {
  // 手指移动超过阈值时取消长按
  longPressActive.value = false
  if (longPressTimer) {
    clearTimeout(longPressTimer)
    longPressTimer = null
  }
}

function handleContextMenu(dish) {
  if (!dish.soldOut) openDishMenu(dish)
}
</script>
```

**优先级**: 🟠 移动端体验核心，不处理会导致功能不可用

```typescript
      if (result === false) return
      if (typeof result === 'string') {
        // 可以 toast 错误信息
        return
      }
    }
    // ... 原有逻辑
  }
}
```

### P1-6: 自动重置定时任务使用 `setInterval` 不可靠

**问题描述**:

```typescript
// server/src/jobs/resetSoldOut.ts
setInterval(async () => {
  const now = new Date()
  if (now.getHours() === 4 && now.getMinutes() === 0) {
    // ...
  }
}, 60_000)
```

**风险场景**:
- 服务器在 03:59 进入休眠/挂起状态
- 04:05 唤醒，`setInterval` 会立即累积执行多次（如果中间错过了多次检查点）
- 但更关键的是：如果服务器在 03:59 - 04:01 之间重启，任务永远不会触发
- 如果 Node.js 进程崩溃后由 PM2 重启，`lastRunDate` 内存变量丢失，可能会重复执行

**修正方案**:

```typescript
// 使用"记录上次执行时间 + 检查跨天"的可靠模式
import { existsSync, readFileSync, writeFileSync } from 'fs'
import { join } from 'path'

const STATE_FILE = join(process.cwd(), '.soldout-reset-state')

function getLastRunDate(): string {
  try {
    if (existsSync(STATE_FILE)) {
      return readFileSync(STATE_FILE, 'utf-8').trim()
    }
  } catch { /* ignore */ }
  return ''
}

function setLastRunDate(date: string) {
  try {
    writeFileSync(STATE_FILE, date)
  } catch { /* ignore */ }
}

export function startSoldOutResetJob() {
  setInterval(async () => {
    const now = new Date()
    const today = now.toISOString().slice(0, 10)
    const lastRun = getLastRunDate()

    // 今天已经执行过，跳过
    if (lastRun === today) return

    // 未到 04:00，跳过
    if (now.getHours() !== 4) return

    // 执行重置
    try {
      const pb = getPocketBase()
      const soldOutDishes = await pb.collection('dishes').getFullList({
        filter: 'soldOut = true',
      })
      // ...
      setLastRunDate(today)
      console.log(`[AutoReset] ${soldOutDishes.length} 道菜品已自动恢复`)
    } catch (err) {
      console.error('[AutoReset] 失败:', err)
      // 失败不写入 lastRun，下次会重试
    }
  }, 60_000)
}
```

**优先级**: 🟠 小型系统下概率低，但修正成本极低（10 行代码）

### P1-7: 迁移文件命名不规范

**问题描述**:

方案中迁移文件命名为 `pb_migrations/1779xxxx_add_soldOut_to_dishes.js`。

PocketBase 的迁移机制要求文件名以**Unix 时间戳**开头（10 位数字），例如 `1713590400_add_soldOut_to_dishes.js`。PocketBase 按文件名排序执行迁移，如果时间戳格式不对，迁移不会被执行。

**修正方案**:

```bash
# 生成正确的迁移文件名
MIGRATION_NAME="$(date +%s)_add_soldOut_to_dishes.js"
echo "迁移文件名: ${MIGRATION_NAME}"
# 输出示例: 1713590400_add_soldOut_to_dishes.js
```

文档中的示例代码应改为：
```bash
# 生成迁移文件
# Linux/macOS
MIGRATION_FILE="pb_migrations/$(date +%s)_add_soldOut_to_dishes.js"
```

**优先级**: 🟠 会导致迁移不执行，功能无法上线

## 四、P2 - 业务场景遗漏

### P2-1: 顾客端弹窗"替换/移除"缺少"替换推荐"逻辑

**问题描述**:

方案中提到顾客端：
```typescript
if (soldOutItems.length > 0) {
  showReplaceDialog(soldOutItems)  // 允许顾客选择替换菜品或移除
  return
}
```

但没有说明**推荐什么菜品作为替换**。顾客面对"铁锅鱼已沽清"时，系统应该推荐同分类的类似菜品（如铁锅鸡），而不是让顾客自己翻菜单找。

**建议方案**:

```typescript
function findReplacement(soldOutDish: Dish, allDishes: Dish[]): Dish | null {
  // 1. 同分类、价格最接近的菜品
  const sameCategory = allDishes.filter(
    (d) => d.category === soldOutDish.category && !d.soldOut && d.id !== soldOutDish.id
  )
  if (sameCategory.length === 0) return null

  // 按价格差排序
  return sameCategory.sort((a, b) =>
    Math.abs(a.price - soldOutDish.price) - Math.abs(b.price - soldOutDish.price)
  )[0]
}
```

**优先级**: 🟡 增强体验，但不阻塞核心功能

### P2-2: 已创建订单中的菜品被沽清后的处理

**问题描述**:

方案没有明确说明：
- 订单已创建（状态为 pending/cooking），订单中的某道菜被标记为沽清
- 此时顾客要求"再加一份这道菜"，如何处理？
- 厨房大屏是否应该对已有订单中的 soldOut 菜品做特殊标记？

**当前行为分析**:
- `PublicOrderAPI.appendOrderItems` 没有 soldOut 校验（P1-4）
- Hook `onRecordBeforeUpdateRequest` 会拦截（但性能差，P0-3）
- 厨房大屏不显示菜品 soldOut 状态

**建议方案**:

1. **追加菜品时**：前端 + Hook 双重拦截，提示"该菜品已沽清，无法追加"
2. **厨房大屏**：已有订单中的 soldOut 菜品用黄色警告标签标注（"原料不足"），但仍需制作（已承诺顾客的不能取消）
3. **订单详情页**：已下单的 soldOut 菜品显示警告标签，提示服务员与顾客沟通

**优先级**: 🟡 业务完整性问题

### P2-3: `stockQty` 预留字段在当前阶段是 YAGNI

**问题描述**:

方案在迁移文件中增加了 `stockQty` 字段：
```javascript
collection.fields.add(new Field({
  name: 'stockQty',
  type: 'number',
  required: false,
}))
```

但当前方案完全没有使用这个字段的逻辑。小型餐厅的老板需求是"今天鱼没了"，不是"还剩 3 条鱼"。

**YAGNI（You Aren't Gonna Need It）分析**:
- 精确库存管理需要：进货记录、消耗记录、盘点、损耗计算
- 这些功能在当前系统中完全不存在
- 增加 `stockQty` 字段会让 Dish 接口变得混乱，前端需要处理这个永远为空的字段
- 如果未来真的需要精确库存，再做一次迁移即可（PocketBase 迁移成本极低）

**建议**:
移除 `stockQty` 字段，保持数据模型精简。在 `Dish` 接口的注释中预留说明即可：
```typescript
export interface Dish {
  // ...
  soldOut?: boolean
  // 未来如需精确库存管理，可在此扩展 stockQty 字段
}
```

**优先级**: 🟡 建议移除，保持模型简洁

## 五、P3 - 建议优化（不阻塞开发）

### P3-1: `escapePbString` 的防御范围不足

```typescript
export function escapePbString(value: string): string {
  value = value.replace(/[|&<>#]/g, '')
  return value.replace(/'/g, "''")
}
```

当前只处理了 `|&<>#'`，但 PocketBase 的 filter 语法中 `~`（正则）、`@`（日期函数）也可能被利用。建议增加：
```typescript
value = value.replace(/[~@$%^*()\[\]{}\\]/g, '')
```

### P3-2: `dish.schema.ts` 未更新

```typescript
// src/schemas/dish.schema.ts
export const dishFormSchema = z.object({
```

虽然 `soldOut` 不在表单中直接编辑（通过 toggle API），但如果有管理后台的菜品编辑表单，Zod schema 应该同步。

### P3-4: 回滚方案遗漏 public-api 容器的重建

如果 public-api 是 Docker 部署，回滚时需要重新构建镜像：

```bash
docker compose up -d public-api
```

文档中只提到了 systemd 服务的回滚，需要补充容器化部署的回滚方式。

### P3-5: 缺少前端加载时的 `soldOutAt` 日期兜底

方案提到"前端加载时检查 soldOutAt 日期，跨天后自动恢复"，但没有给出具体实现。建议在 `OrderFormView` 的 `loadDishes` 中增加：

```typescript
function cleanupExpiredSoldOut(dishes: Dish[]): Dish[] {
  const today = new Date().toISOString().slice(0, 10)
  return dishes.map((d) => {
    if (d.soldOut && d.soldOutAt) {
      const soldOutDate = d.soldOutAt.slice(0, 10)
      if (soldOutDate !== today) {
        // 跨天的沽清自动恢复（服务端定时任务的兜底）
        return { ...d, soldOut: false, soldOutNote: '', soldOutAt: undefined }
      }
    }
    return d
  })
}
```
