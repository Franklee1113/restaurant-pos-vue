## 1. 系统概述

### 1.1 系统定位

智能点菜系统是一套面向餐饮行业的轻量化订单管理解决方案，采用前后端分离架构，支持订单创建、状态流转、菜品维护和数据统计等核心功能。

**2026-04-13 重大更新**：前端已完成从原生 JavaScript SPA 到 **Vue 3 + Vite + Pinia + Tailwind CSS + TypeScript** 的全面重构，并已稳定部署上线。

**2026-04-19 重大更新**：
- 订单状态机升级：新增 `settled`（已结账）状态，将「上菜完成」与「结账离店」拆分为两个独立阶段
- 清台逻辑优化：`completed` 不再自动清台，增加 completed 订单手动清台阻断
- 订单列表新增「桌台状态」独立列，直观显示桌台是否已释放
- `table_status` 集合修复重复数据并增加 `tableNo` 唯一索引

## 4. 核心业务逻辑

### 4.1 订单状态流转

```
                    ┌─────────────┐
                    │   PENDING   │ ◄── 新建订单默认状态
                    │   (待确认)   │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               │               ▼
    ┌─────────────┐        │        ┌─────────────┐
    │   COOKING   │        │        │  CANCELLED  │
    │   (制作中)   │◄───────┘        │   (已取消)   │
    └──────┬──────┘                 └─────────────┘
           │                              ▲
           │                              │
           ▼                              │
    ┌─────────────┐                       │
    │   SERVING   │───────────────────────┤
    │   (待上菜)   │      取消             │
    └──────┬──────┘                       │
           │                              │
           ▼                              │
    ┌─────────────┐                       │
    │  COMPLETED  │───────────────────────┘
    │ (上菜完成)   │
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │   SETTLED   │ ◄── 结账完成/顾客离店
    │  (已结账)   │
    └─────────────┘
```

**状态流转规则**（`utils/orderStatus.ts`）

| 当前状态 | 允许流转到 |
|----------|-----------|
| pending (待确认) | cooking, cancelled |
| cooking (制作中) | serving, cancelled |
| serving (待上菜) | completed |
| completed (上菜完成) | settled |
| settled (已结账) | - (终止状态) |
| cancelled (已取消) | - (终止状态) |

### 4.2 订单金额计算（重构后）

```typescript
// utils/security.ts
export class MoneyCalculator {
  static calculateWithDiscount(
    items: CartItem[],
    discountValue: number,
    discountType: 'amount' | 'percent'
  ) {
    const total = items.reduce((sum, item) => {
      return sum + item.price * item.quantity
    }, 0)

    let discount = 0
    if (discountType === 'percent') {
      discount = Math.round(total * (discountValue / 100) * 100) / 100
    } else {
      discount = discountValue
    }

    const final = Math.max(0, Math.round((total - discount) * 100) / 100)
    return { total, discount, final }
  }
}
```

### 4.3 清台规则

**清台触发条件**（后端 Hook `onRecordAfterUpdateRequest`）

| 订单状态 | 是否自动清台 | 说明 |
|----------|-------------|------|
| `settled` (已结账) | ✅ 自动清台 | 顾客已结账离店，释放桌位 |
| `cancelled` (已取消) | ✅ 自动清台 | 订单取消，释放桌位 |
| `completed` (上菜完成) | ❌ **不清台** | 顾客还在用餐/未结账，桌台保持占用 |

**手动清台前置检查**（前端 `clearTable()`）

```
① 查 table_status → idle ? 阻断提示「无需重复清台」
② 查未完成订单(pending/cooking/serving) → 有 ? 阻断提示「请先处理完毕」
③ 查当前订单状态 = completed → 阻断提示「已上菜完成但尚未结账，请先结账」
④ 以上均通过 → 确认清台 → 执行 API
```

**table_status 数据一致性保障**
- 数据库唯一索引：`idx_table_status_tableNo_unique` 确保一个桌号只有一条记录
- 清台安全校验：后端 Hook 清台前检查 `currentOrderId === record.id`，防止误清其他订单的桌台

## 8. 安全设计

### 8.1 认证机制

1. **JWT Token 认证**
   - 登录成功后获取 Token
   - Token 存储于 `localStorage`（由 `auth.store.ts` 统一管理）
   - 每次 API 请求通过 PocketBase SDK 自动携带 Bearer Token
   - 路由守卫在页面切换时自动校验登录态

2. **Token 刷新**
   - PocketBase 默认 Token 有效期管理
   - 401 响应触发自动登出并跳转登录页

### 8.2 数据安全

1. **XSS 防护** (`utils/security.ts`)
   - 所有用户输入和动态内容通过 `escapeHtml()` 转义
   - 避免将不可信数据直接注入 DOM

2. **金额安全** (`MoneyCalculator`)
   - 使用整数/定点数思维处理金额计算
   - 避免 JavaScript 浮点数精度问题导致的价格偏差

3. **PocketBase 权限规则**
   - 当前配置：`listRule: null` (无限制)
   - 生产环境建议配置详细的 `listRule` / `updateRule`

### 8.3 Nginx 安全配置

- `X-Frame-Options: SAMEORIGIN` 防止点击劫持
- `X-Content-Type-Options: nosniff` 防止 MIME 嗅探
- CORS 跨域配置限制在 `/api/` 路径

### 14.6 状态机升级与清台优化 (2026-04-19)

**改造背景**：
- 旧系统中 `completed`（已完成）状态同时承担「上菜完成」和「顾客离店」两个语义，一旦标记完成就自动清台
- 真实餐饮场景中，上菜完成后顾客通常还在用餐/未结账，此时清台会导致桌位被错误释放
- 缺少「已结账」这一经营终态，营业额统计包含大量未收款订单，数据失真

**改造内容**：

1. **新增 `settled`（已结账）状态**

│  ├── /             → 订单列表                                 │

**关键洞察**：

### 15.4 API 层 (`pocketbase.ts`) — 系统的"血管"

#### 15.4.1 架构分层

```
┌────────────────────────────────────────────────────────────┐
│  应用层视图                                                  │
├────────────────────────────────────────────────────────────┤
│  OrderAPI        │  DishAPI        │  SettingsAPI           │
│  (订单 CRUD)      │  (菜品 CRUD)     │  (设置管理)             │
├────────────────────────────────────────────────────────────┤
│  PublicOrderAPI  │  TableStatusAPI  │                        │
│  (公开无认证)     │  (桌位状态同步)   │                        │
├────────────────────────────────────────────────────────────┤
│  基础设施层                                                  │
│  fetchWithTimeout → handleResponse → getAdminToken          │
│  (超时控制)       │  (统一错误处理)   │  (本地 Token 读取)      │
└────────────────────────────────────────────────────────────┘
```

#### 15.4.2 关键设计洞察

| 设计 | 评价 |
|------|------|
| **`APIError` 自定义异常** | ✅ 统一错误类型，便于 UI 层识别和展示 |
| **`fetchWithTimeout` (AbortController)** | ✅ 30秒超时避免请求挂死 |
| **`sessionExpired` 全局标志** | ✅ 防止 401 时重复弹窗和跳转 |
| **`escapePbString` 转义** | ✅ 防御 PocketBase filter 注入攻击 |
| **`validateTableNo` 正则校验** | ✅ 只允许字母、数字、中文、连字符 |
| **PublicOrderAPI 独立暴露** | ✅ 顾客端点餐无需认证，与管理员 API 分离，最小权限原则 |

#### 15.4.3 潜在风险

1. **认证耦合过深**：几乎每个 API 方法都内联了 `getAdminToken()` 和 401 检查，可以通过请求拦截器统一处理

2. **`PublicOrderAPI.appendOrderItems` 完全在前端做合并**：虽然前端做了状态检查和合并逻辑，但并发场景下两个顾客同时加菜可能产生竞态条件（Racing）。这需要后端做原子性保证，当前仅靠 PocketBase 的单行 PATCH 有一定保护，但非完全安全。

3. **`OrderAPI.updateOrderItemStatus` 的"读-改-写"模式**：
   ```ts
   const order = await this.getOrder(id)
   const items = order.items.map(...)
   return this.updateOrder(id, { items })
   ```
   这不是原子操作，高并发下会丢失更新。需要依赖后端钩子的重算来弥补，但前端应给出更明确的冲突提示。

### 15.5 订单核心模块

#### 15.5.1 订单列表 (`OrderListView.vue`)

**架构特征**：**重逻辑视图组件**，承担了数据获取、筛选、统计、导出、清台、状态流转等大量职责。

**设计亮点**：

| 特性 | 实现 | 评价 |
|------|------|------|
| **智能静默刷新** | `useAutoRefresh` 30秒轮询 + `lastOrderIds` 指纹对比 | ✅ 只在新订单/新结账时播放提示音，避免无谓刷新闪烁 |

| **骨架屏** | `SkeletonBox` 组件 | ✅ 感知加载速度提升 |

   ```ts
   filters.push(`status='${filter.value.status}'`)
   ```
   虽然有 `sanitizePbLike` 做字符白名单过滤，但 filter 拼接逻辑分散在视图层，维护成本高。

#### 15.5.2 订单表单 (`OrderFormView.vue`)

**核心模式**：**创建/编辑复用 + 购物车驱动**

**数据流**：
```
菜品列表 (DishAPI) → 分类筛选 → 点击加菜 → cart 响应式数组
                                    ↓
                            CartPanel 组件 (折扣/餐具/合计)
                                    ↓
                            Zod Schema 校验 → 提交
```

**设计亮点**：

1. **配菜规则引擎 (`DISH_RULES`)**：
   ```ts
   '铁锅鱼': { add: '锅底', qty: 1 }
   ```
   自动关联加菜，减少服务员操作步骤。这是餐饮业务的**领域知识**在代码中的直接体现。

2. **金额计算双保险**：
   - 前端：`MoneyCalculator.calculateWithDiscount`（分转元，防浮点）
   - 后端：钩子强制重算（不信任前端金额）

3. **编辑模式状态隔离**：编辑时不允许直接修改订单状态（提示"请在详情页修改"），防止状态机混乱。

**风险**：

- `hotDishes` 硬编码了招牌菜排序优先级，这是业务规则侵入代码。未来应移到 settings 或数据库配置
- `discountType` 切换时（金额 → 百分比）的自动修正逻辑 `if (discountValue > 10) discountValue = 8` 是"魔法数字"

#### 15.5.3 订单详情 (`OrderDetailView.vue`)

**架构定位**：**只读展示 + 状态操作中心**

**关键洞察**：

1. **状态流转可视化**：右侧"状态操作"面板只显示 `StatusFlow` 允许的下一个状态，形成**强制状态机**，防止非法跳转。

2. **收款码模态框**：集成微信/支付宝二维码展示，支持"听到到账语音后确认"的线下收银流程，贴合小餐馆实际场景。

   - `printKitchenTicket`：后厨单，字体更大、备注更醒目

### 15.11 组合式函数 (Composables)

#### 15.11.1 `useCart` — 购物车核心引擎

**设计模式**：**响应式购物车 + 规则引擎**

```ts
const { cart, cartMap, cartTotalQty, cartTotalAmount, addToCart, ... } = useCart(dishes, DISH_RULES)
```

- `cartMap` 使用 `computed(() => new Map())` 提供 O(1) 的菜品查找
- 数量精度统一用 `Math.round(val * 10) / 10`（支持 0.5 等小数数量）
- **配菜规则自动触发**：`addToCart('铁锅鱼')` → 自动 `addToCart('锅底', 1)`

**测试覆盖**：16 个测试用例覆盖添加、累积、规则触发、数量修改、清空等场景，是项目中**测试最充分的模块**。

#### 15.11.2 `useToast` — 全局通知

- 使用模块级单例 `toasts` 数组，所有组件共享同一通知队列
- 支持 success/error/info/warning 四种类型
- 自动 3 秒后移除

#### 15.11.3 `useConfirm` — 全局确认框

```ts
const ok = await globalConfirm.confirm({
  title: '确认删除',
  description: '此操作不可恢复',
  type: 'danger'

```

- ✅ Promise 化 API，支持 `async/await` 模式
- ✅ 自动处理并发（新 confirm 会关闭旧的并 resolve(false)）

### 15.12 后端钩子 (`pb_hooks/orders.pb.js`) — 业务守护神

这是整个系统中**架构价值最高**的文件之一。它将关键业务逻辑从客户端收归服务端，防止前端篡改。

#### 15.12.1 钩子职责矩阵

| 钩子 | 触发时机 | 职责 |
|------|---------|------|
| `onRecordBeforeCreateRequest` | 创建订单前 | 强制重算金额（不信任前端） |
| `onRecordBeforeUpdateRequest` | 更新订单前 | 检测追加/状态变更 + 重算金额 + 推断整体状态 + table_status 同步 |
| `onRecordAfterCreateRequest` | 创建订单后 | 自动开台（table_status → dining） |
| `onRecordAfterUpdateRequest` | 更新订单后 | 完成/取消时自动清台 |

#### 15.12.2 关键安全设计

1. **防篡改金额**：无论前端传什么 `totalAmount`，后端都根据 `items` 和 `cutlery` 重新计算
2. **防非法追加**：已取消订单不允许追加菜品；已完成订单追加会重置为 pending 并重新开台
3. **状态机守卫**：后端也维护了一套 `flow` 对象，防止绕过前端直接 API 调用造成的非法状态流转
4. **并发安全**：清台时检查 `currentOrderId === record.id`，防止新订单被旧订单的清台操作覆盖

#### 15.12.3 技术债务

- JSON 解析代码在钩子中重复了 4 次，应提取为 `parseJsonField(record, fieldName)` 工具函数
- 所有 `console.log` 都使用字符串拼接，PocketBase 的 JS VM 可能不支持模板字符串的某些特性
