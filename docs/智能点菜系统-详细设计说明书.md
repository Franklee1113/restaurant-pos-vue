## 1. 系统概述

### 1.1 系统定位

智能点菜系统是一套面向小型餐厅的轻量化订单管理解决方案，采用前后端分离架构，支持订单创建、状态流转、菜品维护、营业统计和顾客扫码点餐等核心功能。

系统已完成从原生 JavaScript SPA 到 **Vue 3 + Vite + Pinia + Tailwind CSS + TypeScript** 的全面重构，并稳定部署上线。

#### 项目背景与业务约束

| 维度 | 说明 |
|------|------|
| **目标客群** | 小型餐厅（单店经营，非连锁） |
| **日订单量** | 不足 200 单，属于低并发场景 |
| **人员配置** | 服务人员较少，角色边界模糊 |
| **实际用工** | 服务员、传菜员、老板甚至由同一人兼任 |

**以上背景对后续设计提出的约束**：

1. **权限模型极简**：不强制区分服务员 / 传菜员 / 老板等复杂 RBAC，当前采用「全员管理员」（登录即可操作全部功能）的轻量权限模型，未来如需扩展也仅增加「是否允许改价/折扣」等单点开关，而非多角色矩阵。
2. **交互一站式**：同一操作者可能在「点餐 → 传菜 → 结账 → 清台」之间频繁切换，界面应支持单页面内完成多数操作，减少跨页跳转和重复登录。
3. **容错与兜底优先**：人员少、工作强度大、易疲劳，系统对误操作（重复点击、状态错点、网络抖动）须有充分兜底：后端 Hook 作为唯一可信源、前端操作可撤销、状态非法流转自动阻断。
4. **部署与运维极简**：单服务器 + SQLite 足以支撑日 200 单以下的业务量，避免引入微服务、分布式事务、独立消息队列等过度设计。一键部署脚本 + systemd 守护即满足生产要求。

## 4. 核心业务逻辑

### 4.1 订单状态流转

```
                    ┌─────────────┐
                    │   PENDING   │ ◄── 新建订单默认状态
                    │   (待确认)   │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               │               ▼
    ┌─────────────┐        │        ┌─────────────┐
    │   COOKING   │        │        │  CANCELLED  │
    │   (制作中)   │◄───────┘        │   (已取消)   │
    └──────┬──────┘                 └─────────────┘
           │
           ▼
    ┌─────────────┐
    │   SERVING   │
    │   (上菜中)   │
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │   DINING    │
    │  (用餐中)    │
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │  COMPLETED  │ ◄── 结账完成
    │  (已结账)    │
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │   SETTLED   │ ◄── 清台完成/顾客离店
    │  (已清台)    │
    └─────────────┘
```

**状态流转规则**

| 当前状态 | 允许流转到 |
|----------|-----------|
| pending (待确认) | cooking, cancelled |
| cooking (制作中) | serving, cancelled |
| serving (上菜中) | dining, cancelled |
| dining (用餐中) | completed, cancelled |
| completed (已结账) | settled |
| settled (已清台) | - (终止状态) |
| cancelled (已取消) | - (终止状态) |

### 4.2 清台规则

**自动清台**（后端 Hook `onRecordAfterUpdateRequest`）

| 订单状态 | 是否自动清台 | 说明 |
|----------|-------------|------|
| `settled` (已清台) | ✅ 自动 | 顾客已离店 |
| `cancelled` (已取消) | ✅ 自动 | 订单取消 |
| `completed` (已结账) | ❌ 不清台 | 客人还在，需手动清台 |
| `dining` (用餐中) | ❌ 不清台 | 未结账 |

**手动清台前置检查**

1. 查 table_status → idle ? 阻断「无需重复清台」
2. 查未完成订单 (pending/cooking/serving/dining) → 有 ? 阻断「请先处理完毕」
3. 查当前订单 = dining → 阻断「客人还在用餐中，请先结账」
4. 查当前订单 = completed → 允许清台，同步将订单状态更新为 settled

**清台按钮显示规则（2026-04-21 追加）**

| 订单状态 | 是否显示清台按钮 | 说明 |
|----------|-----------------|------|
| `completed` | ✅ 显示 | 已结账，待清台 |
| `pending`/`cooking`/`serving`/`dining` | ❌ 不显示 | 未完成订单，不可清台 |
| `settled`/`cancelled` | ❌ 不显示 | 终态，已自动清台 |

**异常处理原则（2026-04-21 重构）**
- 订单查询失败 → **阻断清台**，提示"无法确认订单状态"
- `completed→settled` 更新失败 → **阻断清台**，提示"订单状态更新失败"
- 禁止静默忽略错误继续清台，防止 `completed` 订单未归档就释放桌位

### 4.4 订单编辑规则

**编辑权限按状态划分**

| 订单状态 | 是否可编辑 | 说明 |
|----------|-----------|------|
| `pending` (待确认) | ✅ 可编辑 | 未进入厨房制作 |
| `cooking` (制作中) | ✅ 可编辑 | 厨房正在制作 |
| `serving` (上菜中) | ✅ 可编辑 | 服务员正在上菜 |
| `dining` (用餐中) | ✅ 可编辑 | 上菜完成，待结账 |
| `completed` (已结账) | ❌ **不可编辑** | 已收款，财务闭环 |
| `settled` (已清台) | ❌ **不可编辑** | 客人已离店，终态 |
| `cancelled` (已取消) | ✅ 可编辑 | 订单取消后仍可修改记录 |

**阻断实现**
- 前端：`OrderDetailView.vue` / `OrderListView.vue` 的 `handleEdit()` / `editOrder()` 函数中判断状态，若为 `completed`/`settled` 则 `toast.error('已结账/已清台订单不可编辑')` 并返回，不进入编辑页
- 前端按钮样式：`completed`/`settled` 状态下"编辑"按钮变为灰色（`bg-gray-300 text-gray-500 cursor-not-allowed`）
- 后端：`pb_hooks/orders.pb.js` 的 `onRecordBeforeUpdateRequest` 中对 `completed`/`settled`/`cancelled` 修改菜品状态抛出 `"订单已结束，不能修改菜品状态"`

**设计 rationale**
- 已结账订单涉及财务闭环，编辑菜品/金额会导致订单数据与实际收款不一致
- 已打印的纸质账单与系统数据不同步
- 如确有特殊情况（结完账客人又要加菜），应新建关联订单或走反结账流程，而非直接修改原订单

**追加菜品规则（2026-04-21 修订）**
- `dining`（用餐中）订单追加新菜品 → **订单状态保持 `dining`**，仅新增菜品 `status = pending`
- `serving`（上菜中）订单追加新菜品 → **订单状态保持 `serving`**，仅新增菜品 `status = pending`
- 后端自动推断的防回退机制确保状态不会从 `dining`/`serving` 回退到 `pending`
- 已取消订单不允许追加菜品

**删除/减少菜品限制（2026-04-21 新增）**
- 编辑订单时，若删除/减少的菜品原 `status` 为 `cooking`/`cooked`/`served`，后端 Hook 抛出 `400` 错误阻断
- 原因：已制作/已上菜的菜品涉及厨房出餐和纸质账单，不允许直接删除
- 如需退菜，应走专门的退菜流程（未来扩展）

### 4.5 特殊业务规则

**铁锅鱼自动加锅底**（`OrderFormView.vue` / `CustomerOrderView.vue`）
```typescript
const DISH_RULES: Record<string, { add: string; qty: number }> = {
  '铁锅鱼': { add: '锅底', qty: 1 },
  '铁锅炖鱼': { add: '锅底', qty: 1 },
}
```
- 自动加锅底前统一检查锅底（`category='餐具'` 或 `name='锅底'`）是否 `soldOut`
- 若锅底已沽清：`toast.warning('配菜 "锅底" 已沽清，无法自动添加')`，不自动添加
- 员工端和顾客端行为一致

### 5.3 订单接口（管理员）

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/collections/orders/records` | 获取订单列表（支持 filter/sort/perPage） |
| GET | `/collections/orders/records/:id` | 获取单个订单 |
| POST | `/collections/orders/records` | 创建订单 |
| PATCH | `/collections/orders/records/:id` | 更新订单 |
| DELETE | `/collections/orders/records/:id` | 删除订单 |

**Order 数据结构**
```typescript
// 订单整体状态（v1.0.3 新增 dining，v1.0.3 拆分 completed/settled）
type OrderStatusValue = 'pending' | 'cooking' | 'serving' | 'dining' | 'completed' | 'settled' | 'cancelled'

interface Order {
  id: string
  orderNo: string              // 订单号 OYYYYMMDDNNN
  tableNo: string              // 桌号
  guests: number               // 用餐人数
  status: OrderStatusValue     // 订单状态
  items: OrderItem[]           // 菜品项数组
  totalAmount: number          // 订单金额（分计算）
  discount: number             // 折扣金额
  discountType: 'amount' | 'percent'
  discountValue: number
  finalAmount: number          // 实付金额
  waiter?: string              // 服务员
  remark?: string              // 整单备注
  source?: 'staff' | 'customer' // 来源
  customerPhone?: string
  cutlery?: {
    type: 'free' | 'charged'
    quantity: number
    unitPrice: number
    totalPrice: number
  }
  created: string
  updated: string
}
```

**OrderItem 数据结构**
```typescript
interface OrderItem {
  dishId: string
  name: string
  price: number
  quantity: number
  remark?: string
  status?: 'pending' | 'cooking' | 'cooked' | 'served'
}
```

### 5.4 菜品接口（管理员）

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/collections/dishes/records` | 获取菜品列表 |
| POST | `/collections/dishes/records` | 创建菜品 |
| PATCH | `/collections/dishes/records/:id` | 更新菜品 |
| DELETE | `/collections/dishes/records/:id` | 删除菜品 |

**Dish 数据结构**
```typescript
interface Dish {
  id: string
  name: string
  price: number
  category: string
  description?: string
  soldOut?: boolean      // 是否沽清（默认 false）
  soldOutNote?: string   // 沽清备注（如：约30分钟后恢复）
  soldOutAt?: string     // 沽清时间（ISO 日期字符串）
}
```

**菜品状态变更接口**

| 方法 | 路径 | 描述 | 请求体 |
|------|------|------|--------|
| PATCH | `/collections/dishes/records/:id` | 标记/恢复沽清 | `{ soldOut: true, soldOutNote?: string }` |

**SSE 实时推送**

前端通过 `subscribeToDishes(callback)` 订阅菜品变更事件，共享单例 EventSource 连接，避免多组件重复建立连接。菜品状态变更（如 soldOut）秒级推送至所有已连接客户端。

### 5.5 公共接口（顾客端，无需认证）

由 `server/` 公共 API 服务提供：

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/public/dishes` | 获取所有菜品 |
| GET | `/public/table-status/:tableNo` | 查询桌台状态 |
| GET | `/public/orders/by-table/:tableNo` | 查询该桌未完成订单 |
| POST | `/public/orders` | 顾客创建订单 |
| PATCH | `/public/orders/:id/items?token=xxx` | 顾客追加菜品 |

### 5.6 错误码对照表

| HTTP 状态 | 场景 | 前端处理 |
|-----------|------|---------|
| 200 | 成功 | 正常处理 |
| 204 | 无内容 | 正常处理 |
| 400 | 请求参数错误 / 业务校验失败（如 soldOut 菜品、价格异常） | Toast 提示具体错误 |
| 401 | Token 过期 / 未登录 | 清除 token，跳转登录页 |
| 403 | 权限不足 | Toast 提示「权限不足」 |
| 404 | 资源不存在 | Toast 提示 |
| 408 | 请求超时 | Toast 提示「网络超时，请稍后重试」 |
| 409 | 业务状态冲突（如 soldOut 菜品提交订单） | Toast 提示「菜品已沽清，无法下单」|
| 500 | 服务器内部错误 | Toast 提示「服务器错误」+ Sentry 上报 |

## 7. 部署与运维

### 7.1 生产环境架构

- **前端**: Nginx 静态文件服务 (`/var/www/restaurant-pos`)
- **后端**: PocketBase (`127.0.0.1:8090`)，systemd 守护
- **公共 API**: Node.js Fastify 服务 (`127.0.0.1:3000`，Docker 或 PM2)
- **数据库**: SQLite (`/opt/pocketbase/pb_data/data.db`)

### 7.2 Nginx 配置

配置文件：`/etc/nginx/sites-available/restaurant-pos`

```nginx
server {
    listen 80;
    server_name _;
    root /var/www/restaurant-pos;
    index index.html;
    client_max_body_size 50M;

    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_types text/plain text/css application/json application/javascript;

    # API 代理到 PocketBase
    location /api/ {
        proxy_pass http://127.0.0.1:8090;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # SPA 路由回退
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 安全响应头
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
}
```

### 7.3 Systemd 服务

`/etc/systemd/system/pocketbase.service`

```ini
[Unit]
Description=PocketBase
After=network.target

[Service]
Type=simple
User=root
WorkingDirectory=/opt/pocketbase
ExecStart=/opt/pocketbase/pocketbase serve --http="127.0.0.1:8090"
Restart=always

[Install]
WantedBy=multi-user.target
```

### 7.4 一键部署

```bash
cd /var/www/restaurant-pos-vue
sudo bash scripts/deploy.sh
```

`deploy.sh` 执行流程：
1. `npm run build` 构建前端
2. 备份当前生产环境到 `/var/www/restaurant-pos-backups/pre-<timestamp>`
3. 部署 `dist/` 到 `/var/www/restaurant-pos/`
4. 同步 `pb_hooks/` 到 `/opt/pocketbase/pb_hooks/`
5. 重启 PocketBase 并校验运行状态
6. 重启 Nginx 并校验运行状态
7. 健康检查（HTTP 探活），失败则自动回滚

### 7.5 回滚

```bash
# 查看最新备份
ls -la /var/www/restaurant-pos-backups/

# 执行回滚
sudo rm -rf /var/www/restaurant-pos
sudo cp -r /var/www/restaurant-pos-backups/pre-<timestamp> /var/www/restaurant-pos
sudo systemctl restart nginx
```

- 日志：`/var/log/pocketbase-backup.log`

## 8. 安全设计

### 8.1 认证机制

- JWT Token 认证，Token 存储于 `localStorage`
- 401 响应触发自动登出并跳转登录页
- 路由守卫拦截未认证访问

### 8.2 数据安全

- **XSS 防护**: `utils/security.ts` 提供 `escapeHtml()`，打印模板和 DOM 插入均做转义
- **金额安全**: `MoneyCalculator` 前后端双保险，后端 Hook 强制重算不信任前端金额
- **Filter 注入防护**: `escapePbString()` 过滤操作符并转义单引号
- **PocketBase 权限**: 各集合配置 `@request.auth.id != ''`，要求登录

### 8.3 Nginx 安全

- `X-Frame-Options: SAMEORIGIN`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- Content-Security-Policy 限制资源加载源
