> **目标读者**: 测试工程师、开发工程师、QA

> **版本**: v1.0

## 使用说明

本文档采用「思维导图」式层级结构组织测试案例，每个节点可按需展开为具体测试用例。

格式约定：
- `🔴 P0` — 阻塞级，必须每次回归
- `🟡 P1` — 核心功能，必须覆盖
- `🟢 P2` — 边界/异常，建议覆盖
- `⬜` — 当前无自动化覆盖，需补充
- `✅` — 已有自动化覆盖
- `⏭️` — 待实现/跳过

## 1. 订单生命周期测试

### 1.1 正常状态流转 🔴 P0

```
订单状态流转
├── 新建订单 → pending ✅
│   ├── 员工端新建
│   ├── 顾客端新建
│   └── 验证: 桌台自动变为 dining
├── pending → cooking 🔴 P0
│   ├── KDS 点击"开始制作"
│   ├── 验证: 单品 pending → cooking
│   └── 验证: 订单状态自动推断为 cooking
├── cooking → serving 🔴 P0
│   ├── KDS 点击"已完成" (cooked)
│   ├── 服务员确认"已上菜" (served)
│   └── 验证: 订单状态自动推断为 serving
├── serving → dining 🔴 P0
│   ├── 所有单品标记为 served
│   └── 验证: 订单状态自动推断为 dining
├── dining → completed 🔴 P0
│   ├── 员工点击"结账完成"
│   └── 验证: 桌台仍 dining，不清台
└── completed → settled 🔴 P0
    ├── 员工点击"清台"
    ├── 验证: 订单状态变为 settled
    └── 验证: 桌台自动变为 idle
```

### 1.2 取消订单流程 🔴 P0

```
取消订单

│   └── 验证: 桌台自动变为 idle
├── cooking 状态取消
│   ├── 验证: 订单变为 cancelled
│   └── 验证: 桌台自动变为 idle
├── serving 状态取消
│   ├── 验证: 订单变为 cancelled
│   └── 验证: 桌台自动变为 idle
├── dining 状态取消
│   ├── 验证: 订单变为 cancelled
│   └── 验证: 桌台自动变为 idle
├── completed 状态取消 ⬜ 🟢 P2

```
状态回退防护

├── serving → cooking ⬜

## 2. 金额计算测试

### 2.1 基础计算 🔴 P0

```
金额计算
├── 整数价格菜品
│   ├── 1份 ¥68 = ¥68

└── 空购物车边界 ⬜ 🟢 P2
    └── 验证: 提交前被表单校验拦截
```

### 2.2 折扣计算 🟡 P1

```
折扣规则
├── 百分比折扣 (8折)
│   ├── 菜品小计 ¥100，8折 → ¥80，折扣 ¥20
│   ├── 菜品小计 ¥100，9.5折 → ¥95，折扣 ¥5 ⬜
│   ├── 边界: 10折 → 无折扣
│   ├── 边界: 0折 → 非法输入（前端校验拦截）
│   └── 边界: >10折 → 前端自动重置为8
├── 固定金额减免
│   ├── 菜品小计 ¥100，减免 ¥20 → ¥80
│   ├── 减免 ¥0 → 无折扣
│   └── 边界: 减免 > 菜品小计 → final = 0（不小于0）
├── 折扣 + 餐具费 🔴 P0
│   ├── 菜品 ¥100，餐具费 ¥8，8折
│   ├── 正确: 折扣仅作用于菜品 → ¥80 + ¥8 = ¥88
│   └── 错误: 折扣作用于 (菜品+餐具) → 待验证是否被阻断
└── 前后端一致性 🔴 P0
    ├── 前端计算结果
    ├── 后端 Hook 重算结果
    └── 验证: 二者必须一致（允许 ±0.01 浮点误差）
```
