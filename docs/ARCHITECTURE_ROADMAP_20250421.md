## 2. 优先级定义

| 优先级 | 定义 | 处理时限 | 不处理的后果 |
|--------|------|---------|-------------|
| **P0 - 紧急** | 安全漏洞、数据丢失风险、生产稳定性威胁 | 1 周内 | 可能直接导致数据泄露、财务损失、系统不可用 |
| **P1 - 重要** | 架构债务、可维护性瓶颈、核心业务逻辑缺陷 | 1 个月内 | 需求迭代变慢、Bug 率上升、重构成本指数增长 |
| **P2 - 改进** | 工程化提升、质量保障、开发效率优化 | 2-3 个月 | 团队生产力停滞、新人上手困难 |
| **P3 - 规划** | 长期演进、功能扩展、技术栈升级 | 3-6 个月 | 错失业务扩展窗口 |

## 3. P0 - 紧急安全与稳定性（必须立即处理）

### P0-1: 生产环境启用 HTTPS

| 属性 | 内容 |
|------|------|
| **问题描述** | 生产环境 `http://43.143.169.88/` 使用明文 HTTP，JWT Token、订单数据、支付二维码全部明文传输 |
| **风险等级** | 🔴 严重 |
| **影响范围** | 所有用户、所有订单数据、管理员凭证 |
| **根因分析** | 部署时未配置 TLS 证书 |
| **建议方案** | 1. 申请 Let's Encrypt 免费证书（certbot）<br>2. Nginx 配置 80→443 强制跳转<br>3. HSTS 响应头<br>4. 更新所有文档中的 `http://` 为 `https://` |
| **预估工作量** | 0.5 人天 |
| **验收标准** | 1. 全站仅可通过 HTTPS 访问<br>2. SSL Labs 评分 ≥ A<br>3. `http://` 访问自动 301 跳转 |

| **验收标准** | 1. 生产密码已修改且复杂度 ≥ 12 位<br>2. `git log --all --grep="REDACTED_DEFAULT_PASSWORD"` 无结果<br>3. 所有文档无明文密码 |

## 4. P1 - 架构债务治理（1 个月内）

### P1-1: pb_hooks 核心逻辑外迁（最关键架构决策）

| 属性 | 内容 |
|------|------|
| **问题描述** | `pb_hooks/orders.pb.js` 533 行承载了金额计算、状态机、清台同步、沽清校验等核心业务，但运行在 goja VM 中：无法复用代码（函数必须内联）、无法单元测试、无法类型检查、调试只能靠 console.log |
| **风险等级** | 🔴 高 |
| **影响范围** | 订单核心业务流程 |
| **根因分析** | PocketBase Hook 的设计定位是"轻量级校验"，不应承载复杂业务引擎 |
| **建议方案** | **阶段 1（立即）**: 将金额计算、状态机推断、沽清校验提取为纯函数，放入 `server/src/core/`（Node.js 服务可测试、可复用）<br>**阶段 2（后续）**: PB Hook 只保留轻量级触发器，核心逻辑通过 HTTP 调用 Node.js 服务，或迁移到独立后端 |
| **预估工作量** | 3-5 人天 |
| **验收标准** | 1. 金额计算和状态机推断有 ≥ 80% 的单元测试覆盖<br>2. `pb_hooks` 行数降至 200 行以内<br>3. 全量订单流程回归通过 |

### P1-2: API 层拆分（消除 God Object）

| 属性 | 内容 |
|------|------|
| **问题描述** | `src/api/pocketbase.ts` 762 行，被 23 个文件直接导入，混合了 HTTP 客户端、类型定义、缓存、业务校验、6 个领域 API、SSE 连接管理 |
| **风险等级** | 🟡 高 |
| **影响范围** | 全项目数据访问层 |
| **建议方案** | 按以下结构拆分：<br>```<br>src/api/<br>  client.ts          # fetchWithTimeout, handleResponse, 拦截器<br>  cache.ts           # MemoryCache<br>  types/             # Order, Dish, TableStatus 等 DTO<br>  services/<br>    order.service.ts<br>    dish.service.ts<br>    table.service.ts<br>    settings.service.ts<br>    stats.service.ts<br>  realtime/<br>    sse.ts<br>    orders.channel.ts<br>    dishes.channel.ts<br>``` |
| **预估工作量** | 2-3 人天 |
| **验收标准** | 1. 无单文件超过 200 行<br>2. 所有原有导入路径通过 index.ts 兼容导出<br>3. 测试全部通过 |

### P1-3: 引入数据层抽象（Repository Pattern）

| 属性 | 内容 |
|------|------|
| **问题描述** | 前端直接拼接 PocketBase 的 REST URL（`/collections/orders/records?page=...`），与 BaaS 深度耦合，未来迁移成本极高 |
| **风险等级** | 🟡 高 |
| **影响范围** | 数据持久层 |
| **建议方案** | 封装 Repository 接口：<br>`OrderRepository.create(data) → Promise<Order>`，内部再映射到 PocketBase URL。未来切换后端时只需替换 Repository 实现 |
| **预估工作量** | 1-2 人天 |
| **验收标准** | 1. 视图中不出现 PocketBase 特定的 URL 拼接逻辑<br>2. 新增 Repository 接口有对应单元测试 |

### P1-4: 订单编辑引入乐观锁

| 属性 | 内容 |
|------|------|
| **问题描述** | 两个服务员同时编辑同一订单，后保存的请求会无感知覆盖先保存的请求。PocketBase PATCH 是记录级替换 |
| **风险等级** | 🟡 高 |
| **影响范围** | 订单并发编辑场景 |
| **建议方案** | 1. 前端获取订单时记录 `updated` 时间戳<br>2. 提交时携带 `originalUpdated` 字段<br>3. PB Hook 中校验：若数据库中 `updated` 已变化，返回 409 Conflict |
| **预估工作量** | 1 人天 |
| **验收标准** | 并发编辑同一订单时，后提交者收到明确冲突提示 |

### P1-5: 引入 TanStack Query（Vue Query）重构数据层

| 属性 | 内容 |
|------|------|
| **问题描述** | 当前所有数据获取都是命令式（手动 fetch + ref），每个视图各自管理 loading/error/缓存，样板代码重复 |
| **风险等级** | 🟡 中 |
| **影响范围** | 全项目数据获取逻辑 |
| **建议方案** | 引入 `@tanstack/vue-query`，替换手动 fetch 模式：<br>```typescript<br>const { data: orders, isLoading, error } = useQuery({<br>  queryKey: ['orders', page, filter],<br>  queryFn: () => OrderAPI.getOrders(page, filter),<br>  staleTime: 5000,<br>})<br>``` |
| **预估工作量** | 3-4 人天 |
| **验收标准** | 1. 消除视图中手动管理的 `loading` / `error` ref<br>2. 自动缓存、重试、去重生效<br>3. 与现有 SSE 推送无冲突 |

### P1-6: 拆分超大视图组件

| 属性 | 内容 |
|------|------|
| **问题描述** | 视图文件过大：CustomerOrderView.vue(889 行)、OrderListView.vue(828 行)、OrderFormView.vue(731 行) |
| **风险等级** | 🟡 中 |
| **影响范围** | 视图层可维护性 |
| **建议方案** | 按职责拆分，例如 `OrderFormView.vue` 拆分为：<br>- `OrderFormView.vue` — 页面容器<br>- `OrderDishSelector.vue` — 菜品选择<br>- `OrderSummaryPanel.vue` — 订单汇总<br>- `OrderFormContainer.vue` — 数据获取与提交 |
| **预估工作量** | 2-3 人天 |
| **验收标准** | 1. 单文件不超过 400 行<br>2. 新组件有独立测试覆盖 |

## 5. P2 - 工程化与质量提升（2-3 个月）

### P2-1: 补齐 views 单元测试至 75%+/80%+

| 属性 | 内容 |
|------|------|
| **问题描述** | views 模块 Functions 覆盖率 63.14%，Branches 66.71%，CustomerOrderView 仅 42.25%。核心用户交互路径（异常处理、条件渲染、复杂弹窗）缺乏测试保护 |
| **风险等级** | 🟡 中 |
| **建议方案** | 详见 `docs/OPTIMIZATION_PLAN_20250421.md` §2.2。按业务风险排序：<br>1. `CustomerOrderView`: 支付流程、加菜弹窗、餐具费一致性、沽清响应、网络异常<br>2. `OrderListView`: 批量筛选、清台阻断、打印弹窗、分页边界<br>3. `OrderFormView`: 菜品搜索过滤、大图预览、编辑回显、折扣边界<br>4. `SettingsView`: 图片上传预览、表单校验失败 |
| **预估工作量** | 2-3 人天 |
| **验收标准** | views Functions 覆盖率 ≥ 70%，Branches ≥ 75%（第一阶段）；最终目标 Functions ≥ 80%，Branches ≥ 80% |

### P2-2: 建立最小权限模型

| 属性 | 内容 |
|------|------|
| **问题描述** | 当前全员管理员，任何登录用户可删除历史订单、修改系统设置、查看营业统计。无操作审计日志 |
| **风险等级** | 🟡 中 |
| **影响范围** | 权限与审计 |
| **建议方案** | 1. 扩展 `users` 集合增加 `role` 字段<br>2. 定义至少 3 个角色：`admin`（全部权限）、`waiter`（下单/改单/查看）、`kitchen`（仅 KDS）<br>3. 前端路由守卫和 API 层增加权限校验<br>4. 关键操作（删单、改价、清台）记录操作日志 |
| **预估工作量** | 3-4 人天 |
| **验收标准** | 1. 不同角色登录后看到不同导航菜单<br>2. 越权操作被后端拦截并返回 403 |

### P2-3: 增加 Staging 环境

| 属性 | 内容 |
|------|------|
| **问题描述** | CI/CD 直接部署 `main` 分支到生产，无灰度过程 |
| **风险等级** | 🟡 中 |

### P2-4: 统一错误码体系

|------|------|

| **预估工作量** | 1-2 人天 |
| **验收标准** | 1. 全项目无字符串匹配错误判断<br>2. 新增错误时只需在一处定义 |

### P2-5: 清理生产环境 console 输出

| 属性 | 内容 |
|------|------|

| **风险等级** | 🟢 低 |
| **建议方案** | 1. Vite 构建配置使用 `drop_console: true`<br>2. 或 ESLint 规则 `no-console` 设为 error |
| **预估工作量** | 0.3 人天 |
| **验收标准** | 生产构建产物中无 console 调用 |

## 6. P3 - 长期演进规划（3-6 个月）

### P3-1: 数据模型范式化（order_items 关联表）

| 属性 | 内容 |
|------|------|
| **问题描述** | `orders.items` 以 JSON 数组存储，无法高效索引和统计。随着订单量增长（年 7 万+），查询性能将断崖式下降 |
| **风险等级** | 🟡 中（当前低，未来高） |
| **影响范围** | 数据库设计与查询性能 |
| **建议方案** | 引入 `order_items` 独立集合：<br>```<br>order_items { id, orderId, dishId, name, price, quantity, status, remark, created, updated }<br>```<br>订单创建时同时写入 order_items 记录，支持单品级统计和索引 |
| **预估工作量** | 5-7 人天 |
| **验收标准** | 1. 新增订单时自动创建 order_items 记录<br>2. "今天卖了多少份铁锅炖"可在 100ms 内返回<br>3. 历史数据迁移完成 |

### P3-2: 后端核心服务独立化

| 属性 | 内容 |
|------|------|
| **问题描述** | PocketBase 同时承担认证、数据库、实时推送、业务 Hook，已成为单点瓶颈 |
| **风险等级** | 🟡 中（当前低，未来高） |
| **建议方案** | 将 `server/` Node.js 服务扩展为独立核心业务层：<br>- 订单服务（状态机、金额计算）<br>- 菜品服务（库存、沽清）<br>- 统计服务（聚合查询）<br>PocketBase 降级为纯数据库 + 认证服务 |
| **预估工作量** | 10-15 人天 |
| **验收标准** | 1. 订单创建/更新走 Node.js 服务，不再依赖 PB Hook<br>2. 核心逻辑有 ≥ 80% 测试覆盖 |

### P3-3: 桌台可视化（大厅布局图）

| 属性 | 内容 |
|------|------|
| **问题描述** | 订单列表以表格展示，无法直观反映大厅桌台占用情况 |
| **风险等级** | 🟢 低 |
| **建议方案** | 新增可视化页面，以网格/座位图展示 15 个桌台状态，支持点击桌台直接下单或查看订单 |
| **预估工作量** | 3-5 人天 |
| **验收标准** | 1. 一眼看出所有桌台状态<br>2. 点击桌台跳转对应订单 |

### P3-4: 打印模板引擎化

| 属性 | 内容 |
|------|------|
| **问题描述** | `printBill.ts` 硬编码 HTML 字符串，无法适配不同尺寸打印机或自定义格式 |
| **风险等级** | 🟢 低 |
| **建议方案** | 引入轻量级模板引擎（如 Handlebars），支持用户在设置页自定义打印模板 |
| **预估工作量** | 2-3 人天 |
| **验收标准** | 1. 打印内容可通过模板配置<br>2. 支持 58mm/80mm 热敏纸适配 |

### P3-5: 深色模式与无障碍支持

| 属性 | 内容 |
|------|------|
| **问题描述** | 当前只有浅色主题，KDS 大屏在暗光环境下刺眼；缺少无障碍支持 |
| **风险等级** | 🟢 低 |
| **建议方案** | 1. Tailwind CSS `dark:` 前缀适配全站<br>2. KDS 页面默认深色模式<br>3. 增加 `prefers-color-scheme` 媒体查询支持 |
| **预估工作量** | 3-5 人天 |
| **验收标准** | 1. 切换深色模式后全站无样式断裂<br>2. KDS 大屏深色模式对比度符合 WCAG AA |
